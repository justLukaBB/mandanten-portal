<objective>
Make the confirmCreditors endpoint respond immediately after saving confirmation to DB — email sending runs asynchronously in the background without blocking the user response.

Purpose: Currently `confirmCreditors` awaits `processClientCreditorConfirmation()` (Zendesk ticket creation, DOCX generation, email sending) and `startMonitoringForClient()` synchronously, blocking the HTTP response for minutes with many creditors. Users see a spinner for the entire duration. The fix: save confirmation first, respond in <2s, then fire email sending as fire-and-forget.

Output: Modified `clientCreditorController.js` with async background processing.
</objective>

**FAST PATH (before response):**

2. Keep UUID generation for legacy creditors without IDs

**SLOW PATH (fire-and-forget after response):**
After sending the response, launch a self-contained async IIFE that runs the email sending work. This block must:
1. Call `this.creditorContactService.processClientCreditorConfirmation(client.aktenzeichen)` — NO await blocking the response
2. Call `this.sideConversationMonitor.startMonitoringForClient()` with the 2-second delay — same as current code
3. Call `this.addZendeskComment()` if applicable — same as current code
4. Wrap everything in try/catch that logs errors with `console.error` — background failures must NOT crash the process
5. Log start and completion of background work for observability: `console.log('🔄 Starting background creditor contact...')` and `console.log('✅ Background creditor contact completed')`

The pattern:
```javascript
// --- Fast path: save confirmation and respond ---
await this.saveClient(client);

res.json({
    success: true,
    message: 'Gläubigerliste erfolgreich bestätigt',
    status: 'creditor_contact_initiated'
});

// --- Slow path: fire-and-forget email sending ---
const aktenzeichen = client.aktenzeichen;
const creditors = client.final_creditor_list || [];

if (creditors.length > 0) {
    (async () => {
        try {
            console.log(`🔄 [Background] Starting creditor contact for ${aktenzeichen}...`);
            const creditorContactResult = await this.creditorContactService.processClientCreditorConfirmation(aktenzeichen);
            console.log(`✅ [Background] Creditor contact completed: ${creditorContactResult.emails_sent}/${creditors.length} emails sent`);

            if (this.sideConversationMonitor) {
                this.sideConversationMonitor.creditorContactService = this.creditorContactService;
                await new Promise(resolve => setTimeout(resolve, 2000));
                const monitorResult = await this.sideConversationMonitor.startMonitoringForClient(aktenzeichen, 1);
                if (monitorResult && monitorResult.success) {
                    console.log(`✅ [Background] Started monitoring ${monitorResult.side_conversations_count} Side Conversations`);
                }
            }

            if (client.zendesk_ticket_id && creditorContactResult.main_ticket_id) {
                await this.addZendeskComment(client, creditors, creditorContactResult);
            }

            console.log(`✅ [Background] All creditor contact tasks completed for ${aktenzeichen}`);
        } catch (bgError) {
            console.error(`❌ [Background] Failed creditor contact for ${aktenzeichen}:`, bgError.message);
        }
    })();
}
```

**IMPORTANT:** The outer try/catch around the entire method stays to handle errors in the fast path (validation, DB save). The fire-and-forget block has its own independent try/catch. Do NOT move the `res.json()` call after the fire-and-forget block — it must come BEFORE.

**Frontend note (no changes needed):** Both `CreditorConfirmation.tsx` and `ConfirmCreditors.tsx` already react to the API response (`response.data.success`), not to email completion data. The `creditor_contact` field in the response was informational but neither component uses it for UI rendering. Removing it from the response is safe.
